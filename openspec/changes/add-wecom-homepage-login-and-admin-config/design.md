## Context

当前仓库已经具备部分企业微信登录基础能力，但能力分布是断裂的：

- 前端登录页 [`web-admin/src/auth/LoginPage.js`](/Users/mt/code/mt-ai/aicodex/aicodex-admin/web-admin/src/auth/LoginPage.js) 仍以密码、验证码、WebAuthn、LDAP、Face ID 和 WeChat 为主，没有像参考项目 `../CRM-Agent` 那样提供明确的企业微信扫码登录页签、扫码容器和反馈状态。
- OAuth 登录跳转构造逻辑已在 [`web-admin/src/auth/Provider.js`](/Users/mt/code/mt-ai/aicodex/aicodex-admin/web-admin/src/auth/Provider.js) 中区分 `WeCom` 的 `Internal` / `Third-party` 与 `Normal` / `Silent` 模式，并已经兼容回调中的 `auth_code`。
- 后端企业微信 IdP 已存在 [`admin/idp/wecom_internal.go`](/Users/mt/code/mt-ai/aicodex/aicodex-admin/admin/idp/wecom_internal.go) 和 [`admin/idp/wecom_third_party.go`](/Users/mt/code/mt-ai/aicodex/aicodex-admin/admin/idp/wecom_third_party.go)，但内部应用链路仍使用旧式用户信息换取接口和字段模型，和当前前端登录入口存在协议错位风险。
- 后台 Provider 编辑页 [`web-admin/src/ProviderEditPage.js`](/Users/mt/code/mt-ai/aicodex/aicodex-admin/web-admin/src/ProviderEditPage.js) 虽然暴露了 `Sub type`、`Method`、`Scope`、`Agent ID` 等字段，但缺少基于企业微信登录场景的字段解释、必填约束、推荐配置和联调提示；部分字段复用了历史命名，如 `disableSsl` 被当作 “Use id as name”，语义不直观。

用户明确希望参考 `../CRM-Agent` 的已实现方案，并在当前主页中加入企业微信扫码登录，同时补齐后台企业微信登录配置。因此本次设计既要覆盖首页交互，也要覆盖运营配置和登录链路的一致性。

## Goals / Non-Goals

**Goals:**

- 在当前 `web-admin` 登录首页中新增明确的企业微信扫码登录入口，并与现有登录方式共存。
- 参考 `../CRM-Agent` 的双入口思路，引入扫码容器、扫码发起、失败反馈和重试机制，但保持当前 React 登录页结构不被整页重写。
- 以企业内部网页登录为主链路，优先交付 `WeCom + Internal + Normal` 首页扫码登录能力。
- 明确后台企业微信登录配置的字段语义、必填约束、推荐模式和联调提示，让管理员可以独立完成配置。
- 统一前端发起方式与后端回调解析逻辑，确保内部应用主链路走通，并对其他模式保留兼容校验。
- 为上线和排障补充回调地址、可信域名、CorpId、AgentId、Secret、Scope 等配置口径。

**Non-Goals:**

- 不在本次设计中重做整个登录页品牌风格或把登录页改造成与 CRM-Agent 完全相同的视觉布局。
- 不扩展企业微信之外的其他 OAuth Provider 交互方式。
- 不引入新的独立会话系统或重写整个认证中心后端架构。
- 不在本次变更中实现企业微信组织架构同步、自动绑号后台或多租户 SaaS 级别的服务商管理台。

## Decisions

### 1. 登录首页采用“现有登录壳 + 企业微信专属入口”方案，而不是重写整张登录页

当前 `LoginPage.js` 已经承担多登录方式切换、表单提交、OAuth 跳转和多种认证模式兼容。如果完全照搬 CRM-Agent 的新页面，会把本次需求扩大成整页 UI 重构，风险和回归范围都会显著变大。

因此本次采用增量方案：

- 保留当前 `LoginPage.js` 作为主壳。
- 在现有登录方式切换中加入企业微信登录入口。
- 为企业微信登录提供专属扫码区域、提示文案和重试动作。
- 未配置或配置不完整时，在登录页上给出明确反馈，而不是静默隐藏或直接跳错。

备选方案是整体改造登录页，完全复刻 CRM-Agent 的双栏布局。该方案视觉效果更统一，但会把本次“加一个企业微信扫码登录能力”升级为“登录页全面重构”，不采用。

### 2. 首页扫码登录优先使用企业微信官方扫码组件，失败时回退到显式发起链接

CRM-Agent 的实现表明，企业微信扫码登录体验最佳的方式是：

- 首页直接渲染企业微信官方扫码组件；
- 同时保留一个显式“发起企业微信扫码登录”按钮，作为脚本加载失败、组件无法渲染或环境不兼容时的回退入口。

在当前仓库中，这意味着：

- React 端新增一个 WeCom 登录面板组件，负责拉取扫码配置、加载官方扫码脚本、渲染二维码容器与错误文案。
- 若扫码组件不可用，则降级为点击按钮直接跳转到 `Provider.getAuthUrl()` 构造出的授权地址。
- 首页需要保留加载态、未配置态、已过期/失败态和刷新/重试能力。

备选方案是始终使用普通跳转按钮，不在页面内嵌二维码。该方案实现最简单，但和“扫码登录”诉求不匹配，也不符合参考项目的用户体验，因此不采用。

### 2.1 本次首页扫码登录主链路限定为 Internal + Normal

当前仓库虽然已经在 `Provider.js` 中保留了 `Internal` / `Third-party` 与 `Normal` / `Silent` 的多个组合，但这几条链路的企业微信接口口径、所需参数和验收方式并不完全一致。若本次把所有组合都纳入首页扫码登录的同等交付目标，实施和联调范围会急剧膨胀。

因此本次明确：

- 首页二维码登录主链路以 `WeCom + Internal + Normal` 为准。
- `Third-party` 与 `Silent` 组合在本次主要完成后台配置梳理、字段校验和错误提示收敛。
- 非主链路组合保留现有跳转能力与兼容性检查，但不承诺本次补齐与首页二维码面板完全等价的体验。

这样做的原因：

- 用户当前诉求更接近“企业内部员工用企业微信扫码进入认证中心”。
- 现有前端 `Normal` 模式已经使用 `login.work.weixin.qq.com` 入口，更适合作为首批可验收链路。
- 后续如果要扩展服务商网页登录，可以在当前配置整理完成后单独追加 change，而不把本次交付拖成大而全。

备选方案是一次性把 `Third-party + Normal` 也纳入首页二维码主链路。该方案不是不能做，但需要更多联调前提和测试资源，不作为当前最小可交付范围。

### 3. 后台配置按“登录模式驱动字段要求”收敛，而不是让管理员自行理解所有组合

当前企业微信 Provider 存在两层选择：

- `subType`: `Internal` / `Third-party`
- `method`: `Normal` / `Silent`

但配置页尚未把这些选择与字段要求清晰绑定。设计上应把后台配置整理成面向登录场景的规则：

- `Internal + Normal`：必须明确要求 `CorpId`、`Secret`、`AgentId`，并展示适用场景为“PC 扫码登录”。
- `Internal + Silent`：必须明确要求 `CorpId`、`Secret`，展示适用场景为“企业微信内静默登录/网页授权”。
- `Third-party + Normal`：必须明确要求服务商应用 `AppId` 与 `ProviderSecret`，展示适用场景为“多企业服务商网页登录”。
- `Third-party + Silent`：仅在确有支持价值时保留，否则在 UI 上标注限制或谨慎提示。

同时，配置页应补充：

- 回调地址展示与复制入口；
- 企业微信后台需配置的可信域名/授权域名说明；
- 推荐 scope 默认值与含义说明；
- 对明显缺失字段的保存前校验。

备选方案是继续沿用通用 Provider 字段编辑模式，只加一点帮助文案。该方案无法显著降低误配率，不采用。

### 4. 后端企业微信登录链路需要与当前前端入口对齐到同一套官方接口口径

前端已经采用 `https://login.work.weixin.qq.com/wwlogin/sso/login` 作为主要网页登录入口，并在回调中兼容 `auth_code`。后端必须和这套入口保持一致，不能继续停留在旧文档口径。

因此本次设计要求：

- 内部应用登录链路按当前企业微信官方网页授权 / 登录授权方式核对 `auth_code -> 用户身份` 的接口调用顺序。
- 第三方应用链路继续保留 `service/get_provider_token` 与 `service/get_login_info`，但要补充错误映射和字段落地口径。
- 登录回调错误必须回传到前端可读状态，而不是只在后端日志中出现。
- 对 `userid` / `open_userid` / `name` / `avatar` 等字段建立稳定映射，避免首页登录成功后用户信息缺失。

备选方案是只做前端入口，不修后端接口。该方案会让首页看起来“支持扫码”，但联调阶段仍然会卡在回调解析，不能接受。

### 5. 以文档化回调和联调指引作为交付物的一部分，而不是把排障知识留在代码里

企业微信登录最常见的问题并不在代码本身，而在域名、回调地址、AgentId、Secret、可信域名和可见范围配置错误。如果这些信息只隐含在代码逻辑里，后续会反复出现“配置看似正确但登录失败”的情况。

因此本次设计把以下内容列为正式交付的一部分：

- 后台配置页中的回调地址展示与提示文案；
- 项目文档中的企业微信登录配置清单；
- 对 Internal / Third-party 的选择建议；
- 对一个域名承载认证中心与 AI 网关时的路径/子域名建议；
- 对典型错误场景的排障口径说明。

备选方案是只在代码注释中说明。该方案对运营和实施团队无效，因此不采用。

## Risks / Trade-offs

- [首页增加企业微信扫码区域后，登录页状态复杂度会上升] → 将企业微信扫码逻辑收敛到独立面板组件，不把所有状态散落在 `LoginPage.js` 主体中。
- [企业微信官方脚本加载失败会导致二维码区域空白] → 保留显式跳转按钮与错误提示，确保脚本失败时仍可发起登录。
- [后台继续复用历史字段命名会增加理解成本] → 在本次范围内先通过标签、提示和校验收敛语义；若后续仍有困扰，再考虑数据模型层面的字段重命名。
- [后端内部应用接口口径若与当前实现不一致，可能暴露兼容性问题] → 先在设计和提案中把这一点明确为必改项，并在实施阶段以真实联调为准做回归。
- [同时支持 Internal / Third-party 与 Normal / Silent 组合会提高测试矩阵] → 首页扫码登录聚焦 `Internal + Normal`，其余组合只做配置校验与兼容性收敛。

## Migration Plan

1. 先固化本次主推配置模式为 `Internal + Normal`，并明确回调地址格式和所需字段清单。
2. 在后台 Provider 编辑页中补充企业微信登录配置校验、字段说明和联调提示。
3. 在前端登录页新增企业微信扫码登录入口与独立扫码面板，接入扫码配置获取、二维码渲染和失败回退。
4. 校正后端企业微信回调解析和用户信息换取逻辑，使其与前端入口完全对齐。
5. 补充文档和回归用例，覆盖首页显示、`Internal + Normal` 扫码登录成功/失败、配置不完整、回调异常和后台保存校验。

回滚策略：

- 若首页扫码登录体验不稳定，可先隐藏扫码面板，仅保留现有企业微信 Provider 跳转入口。
- 若后端企业微信接口调整导致联调失败，可先回滚到现有登录后端实现，同时保留后台配置提示和首页入口代码分支以便后续继续修复。

## Open Questions

- 企业微信登录成功后，系统是否需要区分“首次登录未绑定账号”和“可直接登录”两类页面反馈；若当前仓库没有独立绑号流程，是否先统一落为登录失败提示并保留扩展位？
- 后台配置页是否需要额外提供“配置检测”动作，例如校验回调地址、扫码地址和必填项，而不仅仅是保存前静态校验？
